Chronicles of the
Six Houses (CoSH)

An AI-driven narrative RPG engine — built from scratch in React/TypeScript.

CoSH uses a custom LLM quest-generation pipeline to produce dynamic, branching narrative content across 122 characters, 6 houses, and 43 narrative state flags. The system enforces structured output validation, game-state-aware content routing, and programmatic hallucination prevention — all within a solo-engineered React/TypeScript architecture.

The Challenge

AI-driven narrative generation at scale is hard. When you have 122 characters across 6 factions, dozens of quest lines, and player-state-dependent content, the failure modes multiply fast:

  • Hallucination: The LLM invents NPCs, house names, and places that don't exist in the game world
  • Lore inconsistency: Characters reference events that haven't happened, relationships that don't exist, or houses with wrong geography
  • State drift: Multi-step quest chains lose continuity between steps when the LLM doesn't re-anchor canonical data
  • Scale collapse: Prompt context windows overflow when trying to maintain consistency across dozens of simultaneous NPC arcs

Standard approaches — single-shot generation, loose prompting, post-hoc editing — don't survive contact with a system this complex. This is an engineering problem, not a writing problem.

Architecture Overview

CoSH is built as a five-layer pipeline. Each layer has a single responsibility and feeds cleanly into the next:

Layer 1 — Data Models & Game State

TypeScript interfaces define every entity: Characters, NPCs, Houses, Quests, Events, Flags, Relationships. A centralized React state hook manages the full game state — player choices, relationship values, unlocked NPCs, completed quests, active flags — and propagates changes to all consumers reactively.

Layer 2 — Prompt Construction Engine

A dedicated prompt builder assembling context from game state: NPC roster filtered by unlock conditions, active flags, relationship values, house lore, festival state, and player history. Output is a structured prompt with canonical data locks (character names, house names, locations) preventing hallucination at the source. Two-step generation (narrative prose + structured JSON outcome) separated into distinct calls.

Layer 3 — LLM Generation (API Layer)

Prompt delivered to the configured LLM provider. Response parsed as structured JSON containing both narrative content (the story prose) and mechanical outcomes (stat changes, flag updates, relationship deltas, quest progression). Two-step pipeline: Step 1 generates narrative, Step 2 resolves outcomes — each with its own schema and validation.

Layer 4 — Outcome Processing & Validation

Structured outcome passed through a strict-key allowlist validator. Only whitelisted field names survive. Unknown fields are silently stripped (no error, no crash, no invalid data reaching game state). Numeric fields checked against minimum floors. NPC references cross-referenced against the canonical roster. Hallucinated names caught and corrected or rejected. Flag updates routed to the condition engine.

Layer 5 — UI Rendering

Validated outcomes rendered through React components: the narrative text displayed in-scroll, mechanical effects applied to game state, journal entries created, relationship panels updated. UI is mobile-first with responsive breakpoints, touch-friendly targets, and accessible markup.

Prompt Engineering

The prompt system is the core differentiator between CoSH and a chatbot with a theme. Every prompt enforces five categories of constraint:

1. CANONICAL DATA LOCK
Character names, house names, locations, relationships — hard-coded into the prompt, never inferred by the LLM. Step 2+ prompts re-anchor this data from the registry, not from memory.
2. NAMED CAST FILTER
Only NPCs that pass isNPCActive() (unlock year, alive/dead status, house membership) can appear in scenes. Prevents generational NPCs from appearing before they're born.
3. CONTINUITY RE-ANCHOR
Multi-step quest chains include a canonical re-anchor block on Steps 2-3+ that restates the quest name, location, host house, and cast — preventing the LLM from inventing "House Veridian" three steps in.
4. HOUSE CLARITY NOTES
Each house gets a custom prompt block: sibling age order, parent relationships, cultural norms, geographic facts. Prevents cross-house NPC hallucination (e.g., a Cindralith patriarch appearing in a Veythorne scene).
5. STRUCTURED OUTPUT SCHEMA
Every prompt specifies the exact JSON schema the LLM must return. Field names, types, allowed values, and whether each field is required. The schema IS the contract — the LLM generates to it, and the validator enforces it.

LLM Output Validation

Trusting raw LLM output in a complex game world is how you get a character referencing their dead mother who is alive, or a house war that never happened. CoSH enforces output integrity at three levels:

  • Schema whitelist: OUTCOME_ALLOWED_KEYS — only pre-registered field names survive processing. Everything else is silently stripped. No errors, no crashes, no invalid data reaching game state.
  • Entity cross-reference: Every NPC name in the LLM's output is checked against the canonical roster. Hallucinated names are either corrected (if close to a real NPC) or rejected (if entirely fictional).
  • Flag integrity: Flag updates are type-checked (string vs boolean vs number) and value-checked against allowed values defined in the 43-entry FLAG_REGISTRY. Invalid flag values are dropped before they reach the condition engine.

This validation layer is what makes CoSH reliable enough to generate hundreds of quest steps without manual review. The LLM writes. The validator decides what ships.
